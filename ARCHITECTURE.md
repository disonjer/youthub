# YT_player -- Архитектура и техническое описание

## Обзор

YT_player -- консольный YouTube-клиент, полностью заменяющий браузер.
Рендерит TUI-сетку видео-превью в терминале kitty, воспроизводит видео через
кастомный `ffplay-yt` (форк ffplay из FFmpeg 4.3.9 с IPC-расширениями)
и напрямую общается с внутренними протоколами YouTube -- без yt-dlp в пути воспроизведения.

Точка входа: `.venv/bin/python3.11 grid_demo.py`

---

## Общий конвейер

```
grid_demo.py   (TUI-сетка -- kitty graphics protocol)
      |
      | пользователь нажимает Enter на тайле
      v
bridge_player.py   (оркестратор -- запускает bridge + ffplay-yt, управляет IPC)
      |
      +---> sabr_bridge.mjs   (Node.js -- персистентный SABR-стриминг-мост)
      |        |
      |        +---> po_token.mjs      (PO Token через bgutils-js, без браузера)
      |        +---> pr_fetch.py       (получение playerResponse, ротация стратегий)
      |        +---> youtubei.js       (дешифровка n-параметра)
      |        +---> googlevideo       (SabrStream -- клиент протокола SABR)
      |        |
      |        +---> ffmpeg            (мультиплексирует видео+аудио в matroska)
      |                |
      |                v
      |           /tmp/ytlive_<id>_sN.mkv   (растущий файл на диске)
      |
      +---> ffplay-yt                  (патченный ffplay, читает растущий .mkv)
      |        |
      |        +---> IPC unix-сокет    (POS-отчёты, SEEK_REQ, OPEN, QUIT)
      |
      +---> recs_pipeline.py           (рекомендации для бокового меню)
      +---> sponsorblock.py            (авто-пропуск спонсорских вставок)
      +---> watchstats.py              (пинги истории просмотра через куки)
```

## Модуль за модулем

### 1. `grid_demo.py` -- TUI-сетка (главная точка входа)

Основной цикл приложения. Рисует прокручиваемую сетку YouTube-превью
в терминале kitty через kitty graphics protocol (escape-последовательности,
передающие PNG-изображения прямо в терминал).

**Последовательность запуска:**
1. Загружает домашнюю ленту через `youthub.innertube.InnerTube.home()` (клиент TVHTML5).
2. Берёт случайное видео из ленты и запрашивает его `/next` pivot -- ~30 доп. рекомендаций.
3. Объединяет полки home + pivot, дедуплицирует по `videoId`.
4. Параллельно скачивает до 60 превью (пул из 8 потоков).
5. Вычисляет адаптивную раскладку сетки по размеру терминала (колонки, строки, размер ячейки в пикселях).
6. Входит в цикл чтения клавиш через `youthub.terminal.KeyReader` в raw-режиме.

**Движок раскладки** (`compute_layout`):
- Целевая ширина тайла ~36 ячеек, корректируется по ширине терминала.
- Высота тайла вычисляется из соотношения 16:9 с учётом пиксельного размера ячейки kitty.
- Оставшееся вертикальное пространство распределяется в межстрочные промежутки,
  чтобы сетка заполняла экран целиком.
- Учитываются `top_margin`, `bottom_margin`, `side_margin`, `gutter`.

**Навигация:**
- Стрелки / hjkl: перемещение фокуса (жёлтая рамка из box-drawing символов `┏━┓┃┗━┛`).
- PgUp/PgDn/Home: постраничные прыжки.
- Прокрутка: `scroll_offset_for()` удерживает фокус в видимой области,
  привязывается к границам строк.
- **Бесконечная прокрутка**: `FeedLoader` расширяет ленту в фоновом потоке,
  когда пользователь приближается к концу. Берёт случайное недавнее видео,
  запрашивает его `/next` pivot, добавляет новые уникальные результаты.

**Оверлей поиска** (`SearchOverlay`):
- Клавиша `f`/`а` открывает центрированное поле ввода поверх сетки.
- На Enter выполняет `InnerTube.search(query)`, заменяет ленту результатами.
- Перед отрисовкой удаляет перекрывающиеся тайлы (kitty-изображения рендерятся поверх текста).

**Обновление ленты**: `r`/`к` сбрасывает кэши и заново загружает home + pivot.

**Превью при наведении** (`youthub.preview.PreviewPlayer`):
- Через 3 секунды удержания фокуса на тайле запускает фоновое слайд-шоу.
- Циклически показывает 4 сэмплированных кадра YouTube (mq0..mq3.jpg)
  с PIL-альфа-блендинг кроссфейдами (3 промежуточных кадра на переход, ~14 fps).
- Колбэк кадра передаёт PNG через kitty graphics protocol под `_screen_lock`.

**Воспроизведение видео** (клавиша Enter):
1. Останавливает превью, ставит загрузчик ленты на паузу.
2. Показывает `ConnectingAnimation` -- центрированную Unicode-панель с радужными
   эквалайзер-барами (HSL-раскрашенные `▁▂▃▄▅▆▇█`), braille-спиннером (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`)
   и мерцающей разделительной линией, всё анимировано на ~14 fps.
3. Вызывает `play_video()`, делегирующий в `bridge_player.start_player()`.
4. Через `xdotool` / `wmctrl` ищет и фокусирует окно ffplay-yt (несколько
   попыток для совместимости с тайловыми WM, в частности dwm).
5. По возвращении: очищает stdin, перерисовывает сетку.

**Поддержка раскладок**: все горячие клавиши дублированы для EN и RU QWERTY
(например `q`/`й` -- выход, `r`/`к` -- обновить, `f`/`а` -- поиск).

### 2. `bridge_player.py` -- Оркестратор воспроизведения

Управляет жизненным циклом одного сеанса воспроизведения. Координирует
sabr_bridge.mjs (Node), ffplay-yt, SponsorBlock, боковую панель рекомендаций
и пинги статистики просмотра.

**Архитектура:**
- Запускает sabr_bridge.mjs в отдельной группе процессов (`start_new_session=True`).
- Подключается к управляющему сокету моста (`BridgeControl` через Unix domain socket).
- Отправляет `START_SESSION path=<tmpfile> start_at=0` -- ждёт 512 КБ данных
  прежде чем запустить ffplay-yt.
- Подключается к IPC-сокету ffplay-yt и входит в цикл контроллера.

**Цикл контроллера** (`_controller_loop`):
- Читает события из IPC-сокета ffplay-yt построчно.
- `POS <float>`: обновляет `_last_pos_sec` (текущая позиция воспроизведения).
- `SEEK_REQ_REL <float>`: пользователь нажал стрелки в ffplay-yt.
- `PLAY_VIDEO <id>`: пользователь кликнул тайл в боковой панели -- устанавливает
  `next_video_id`, отправляет `QUIT` в ffplay, внешний цикл grid перезапускается с новым видео.
- `LOAD_MORE_RECS`: бесконечная прокрутка в боковой панели, запускает загрузку продолжения.

**Принятие решения о перемотке** (`_handle_seek_req`):
- Вычисляет, попадает ли целевая позиция в текущий .mkv-файл
  (по оценке байт-в-секунду из размера файла / длительности воспроизведения).
- **Перемотка в пределах файла**: отправляет `SEEK_REL <delta>` в ffplay-yt (мгновенно).
- **Перемотка за пределы файла**: запускает `_restart_at` в отдельном потоке:
  1. Отправляет `START_SESSION path=<new_mkv> start_at=<target>` мосту.
  2. Ждёт 4 МБ данных в новом файле (~20-30 секунд 1080p-буфера).
  3. Удаляет старый .mkv, отправляет `OPEN <new_mkv>` в ffplay-yt.
- Кулдаун: 0.5с между перемотками, чтобы спам стрелками не вызывал каскад перезапусков.
- Блокировка перезапуска предотвращает параллельные перезапуски.

**Интеграция SponsorBlock** (`_check_sponsor_skip`):
- На каждом событии POS проверяет, попал ли playhead в спонсорский сегмент.
- Отправляет `SPONSOR_SKIP <category> <duration>` в ffplay-yt (показывает оверлей).
- Затем перематывает за сегмент, используя ту же логику перемотки.
- Сегменты загружаются один раз на видео, кэшируются 24 часа на диске.

**Логика повторных попыток**: запуск моста, START_SESSION и генерация первых байтов
имеют по одной повторной попытке с убийством и перезапуском моста.

**Очистка**: при `kill()` отправляет QUIT мосту, завершает ffplay, затем
`os.killpg(SIGKILL)` на группу процессов моста, чтобы убить осиротевшие ffmpeg-потомки.

### 3. `sabr_bridge.mjs` -- Node.js SABR-стриминг-мост

Персистентный Node.js-процесс, обслуживающий весь стриминг YouTube. Загружается
один раз на видео; переживает перемотки благодаря управляющему протоколу.

**Инициализация (`initOnce`):**
1. **Получение playerResponse** (три уровня):
   - `fetchWatchPlayerResponse`: вызывает `pr_fetch.py` в цикле (до 24 стратегий),
     ротируя TLS-профили при неудаче. Фоллбэк на Node `fetch`.
   - Кэш PR моста (`cache/pr_<vid>.json`, TTL 12 часов).
   - Устаревший кэш Camoufox-бутстрапа.
2. **Дешифровка n-параметра**: загружает `youtubei.js`, вызывает `player.decipher(rawSabrUrl)`
   для расшифровки подписи и n-параметра в SABR-стриминг-URL.
3. **Генерация PO Token**: вызывает `po_token.mjs` с `visitorData` из PR +
   `videoId` как привязка к контенту. Фоллбэк на кэшированный файл.
4. **Выбор форматов**:
   - Видео: H.264 предпочтительнее VP9 на каждом разрешении (приоритет itag:
     299->303->137->248->136->247->135->134->133). H.264 декодируется в ~2-3 раза быстрее.
   - Аудио: поддержка мультитрека -- предпочитается русская дорожка (`audioTrackId`
     начинающийся с 'ru'), затем `audioIsDefault`, затем любая. Приоритет itag:
     251->250->249->140 (Opus high->med->low, AAC последний).

**Жизненный цикл сессии:**
- `startSession`: создаёт `SabrStream` (из npm-пакета `googlevideo`),
  запускает ffmpeg с видео на fd:3 и аудио на fd:4, мультиплексирует в файл matroska.
  ffmpeg запускается с `-copyts` для сохранения абсолютных PTS.
- Потоки-помпы читают из `videoStream`/`audioStream` (ReadableStream) SabrStream
  и пайпят в дополнительные stdio fd ffmpeg.
- `stopSession`: прерывает SabrStream, закрывает пайпы ffmpeg, убивает при необходимости.

**Управляющий протокол** (Unix domain socket, построчный):
- `START_SESSION path=<file> start_at=<sec>` -- ответ `OK session=<id>` или `ERR <msg>`.
- `STOP_SESSION` -- ответ `OK`.
- `QUIT` -- ответ `OK`, затем `process.exit(0)`.
- `STATUS` -- ответ `OK session=<id|none>`.

### 4. `pr_fetch.py` -- Обход бот-стены YouTube для получения playerResponse

Получает `ytInitialPlayerResponse` через curl_cffi (реальные TLS-отпечатки браузеров).

**Система ротации стратегий:**
- 24 стратегии по 3 осям: TLS-профиль (chrome131, chrome145, firefox133,
  safari184, safari180_ios, chrome131_android, edge101) x транспорт (HTML-скрейпинг
  /watch или POST /youtubei/v1/player для ANDROID_VR, IOS, MWEB, TVHTML5,
  WEB_EMBEDDED_PLAYER) x режим прокси (HTTPS_PROXY из env или напрямую).
- **Липкий выбор**: текущая стратегия держится пока не заблокируют,
  затем переход к следующей живой.
- **Предварительное чередование**: соседние стратегии отличаются и по TLS-семейству,
  И по режиму прокси -- максимальное разнообразие отпечатков.
- **Система раундов**: когда все 24 стратегии мертвы, все воскресают и начинается
  раунд N+1.
- Состояние хранится в `cache/strategy_state.json` (атомарная запись через rename).

**Поток получения:**
1. Прогрев: `GET /` для получения куки `VISITOR_INFO1_LIVE`.
2. `GET /watch?v=<id>` с TLS-профилем стратегии.
3. Регулярка извлекает `ytInitialPlayerResponse` из HTML.
4. Проверка наличия `streamingData` -- если нет, значит бот-стена.
5. При успехе: запускает фоновую телеметрию (скачивание превью для органичности).
6. При неудаче: помечает стратегию мёртвой, сдвигает указатель, код возврата 2.

### 5. `po_token.mjs` -- Генератор PO Token (без браузера)

Чеканит YouTube PO (Proof of Origin) Token через bgutils-js + jsdom.

**Как это работает:**
1. Получает BotGuard-челлендж через InnerTube-эндпоинт `/att/get`.
2. Скачивает BotGuard-интерпретатор JS с CDN Google.
3. Исполняет его внутри jsdom-окна (притворяясь Chrome на youtube.com).
4. Создаёт `BotGuardClient`, запускает `snapshot()` для генерации доказательства.
5. Обменивает доказательство на integrity token через `googleapis.com/GenerateIT`.
6. Использует integrity token для чеканки PO Token, привязанных к `videoId` (content-bound).

**Кэширование**: integrity token кэшируется на 80% его TTL (~48 мин из ~1 часа).
Последующие чеканки переиспользуют кэш (~50мс каждая).

### 6. `sabr.py` -- Чисто-питоновский клиент протокола SABR

Реверс-инженерная реализация протокола Server ABR (SABR) стриминга YouTube.

**Контейнерный формат UMP:**
- Кастомная кодировка varint (ведущие 1-биты в первом байте указывают размер 1-5 байт).
- Типы сообщений: MEDIA_HEADER (20), MEDIA (21), MEDIA_END (22),
  NEXT_REQUEST_POLICY (35), FORMAT_INITIALIZATION_METADATA (39),
  SABR_REDIRECT (40), SABR_ERROR (41), STREAM_PROTECTION_STATUS (55),
  SABR_ACK (58), END_OF_TRACK (59).

**Построение тела запроса** (`build_request_body`):
- Берёт protobuf-шаблон (захваченный при бутстрапе) и патчит:
  - Поле 2 (repeated): ID выбранных форматов (itag + lmt + audio_track_id).
  - Поле 3 (repeated): буферизированные диапазоны по треку.
  - Подполя поля 1: player_width (18), player_height (19), bandwidth (23),
    max_height (59), временные метки playhead (28, 29, 36, 39).

**Цикл SABR** (`SabrClient.run`):
- Бутстрап: POST патченного init body, парсинг UMP-ответа.
- Цикл продолжений: сборка тела запроса с обновлённым состоянием буферов, POST, парсинг.
- Обнаружение треков: каждый MEDIA_HEADER анонсирует новый трек (itag + lmt).
- Сборка буфера: MEDIA-пейлоады записываются по абсолютным байтовым смещениям
  (обработка перекрывающихся диапазонов). Вложенные UMP внутри MEDIA рекурсивно обрабатываются.
- Завершение: все треки закончены или 10 пустых ответов подряд.

### 7. `proto_edit.py` -- Библиотека round-trip для Protobuf

Минимальный парсер/сериализатор protobuf с сохранением порядка полей.
Парсит сырой protobuf в дерево `Message`, позволяет мутировать поля
и ресериализует с сохранением исходного порядка и неизвестных wire type.

Поддерживает wire type: VARINT (0), I64 (1), LEN (2), I32 (5).
LEN-поля автоматически рекурсивно парсятся как вложенные сообщения,
когда эвристика `_looks_like_msg()` проходит (валидные номера полей 1..100000, чистый EOF).

### 8. Пакет `youthub/` -- InnerTube + TUI-примитивы

#### `youthub/innertube.py` -- Клиент InnerTube API
- Представляется как TVHTML5 (User-Agent PlayStation 4, ОС Tizen 8.0).
- OAuth Bearer-аутентификация через Device Authorization Grant.
- Эндпоинты: `browse` (главная, подписки), `search`, `next` (рекомендации),
  `player` (URL для трекинга статистики просмотра).
- HTTP/2 через httpx, авто-обновление протухших access token.
- Локаль: `hl=ru`, `gl=RU` для русскоязычных метаданных.

#### `youthub/auth.py` -- OAuth 2.0 Device Authorization Grant
- Стандартный TV-пейринг: "откройте youtube.com/activate и введите XXXX-XXXX".
- Использует известный `client_id` из Kodi/SmartTubeNext.
- Токены хранятся в `cache/oauth.json` (chmod 0600).
- Авто-обновление с запасом 60 секунд; при провале обновления -- интерактивный пейринг.

#### `youthub/feed.py` -- Парсер ответов InnerTube
- Парсит JSON TVHTML5 (глубоко вложенное дерево рендереров) в плоские датаклассы:
  `Video(video_id, title, channel, views, age, duration, badges, thumbnail_url)`,
  `Shelf(title, videos)`, `Feed(shelves, continuation)`.
- Обрабатывает схемы `tileRenderer` (TVHTML5) и `compactVideoRenderer` (WEB).
- Мультиязычные эвристики для определения просмотров/давности (русский и английский).

#### `youthub/feed_loader.py` -- Движок бесконечной прокрутки
- Оборачивает растущий список `Video` с потокобезопасным доступом.
- Фоновый воркер: берёт случайное недавнее видео, запрашивает его `/next` pivot,
  добавляет ранее невиданные результаты, предзагружает их превью.
- `pause()`/`resume()` вокруг воспроизведения, чтобы избежать борьбы за CPU/трафик.

#### `youthub/graphics.py` -- Kitty Graphics Protocol
- `transmit_and_place()`: отправляет PNG чанками по 4096 байт base64, отображает
  в позиции курсора с указанными размерами в ячейках. Использует `q=2` (тихий), `C=1` (без сдвига курсора).
- `delete_image()`, `delete_all()`: удаление изображений по ID или всех.
- `reset_terminal_modes()`: отключает трекинг мыши, отчёт фокуса, bracketed paste,
  показывает курсор -- защитная очистка после завершения дочерних процессов.

#### `youthub/terminal.py` -- Размер терминала и чтение клавиш
- `size()`: ioctl TIOCGWINSZ для колонок, строк и пиксельного размера ячеек.
- `KeyReader`: контекстный менеджер, переключающий stdin в cbreak-режим.
  Декодирует стрелки, PgUp/PgDn, Home/End, Enter, Esc, Tab, Backspace
  и UTF-8 мультибайтовые последовательности (поддержка кириллицы для RU-раскладки).

#### `youthub/thumbnails.py` -- Кэш превью
- Дисковый кэш: JPEG-файлы в `cache/thumbnails/<videoId>.jpg`.
- In-memory PNG-кэш (JPEG->PNG через PIL, `compress_level=1` для скорости).
- Параллельная предзагрузка: `ThreadPoolExecutor(max_workers=8)`.
- Кадры превью: `mq{0,1,2,3}.jpg` с `i.ytimg.com` для слайд-шоу при наведении.

#### `youthub/preview.py` -- Движок превью при наведении
- Слайд-шоу из 4 кадров с PIL alpha-blend кроссфейдами (3 промежуточных кадра).
- Удержание кадра: 850мс, переход: 200мс.
- Начинает с бленда mq0->mq1, чтобы переход от статического превью был плавным.
  Зацикливается бесконечно, пока фокус не сдвинется.

### 9. `ffplay-yt` -- Патченный FFplay

Кастомная сборка `ffplay` из FFmpeg 4.3.9 с расширениями:
- **Флаг `-ipc <unix_socket_path>`**: открывает Unix domain socket для двусторонней
  связи с bridge_player.py.
- **Исходящие события**: `POS <timestamp>` (периодические отчёты о позиции),
  `SEEK_REQ_REL <delta>` (пользователь нажал стрелки перемотки).
- **Входящие команды**: `SEEK_REL <delta>` (перемотка внутри файла), `OPEN <path>`
  (переключение на новый медиафайл без пересоздания окна), `QUIT` (корректное завершение),
  `SPONSOR_SKIP <cat> <dur>` (показ оверлея пропуска), `RECS_ITEM ...` / `CLEAR_RECS`
  (данные тайлов бокового меню), `PLAY_VIDEO <id>` (передача клика на тайл боковой панели),
  `LOAD_MORE_RECS` (триггер бесконечной прокрутки боковой панели).
- **Боковая панель рекомендаций**: переключается по Tab, рендерит тайлы с превью+текстом,
  полученные по IPC. Клик на тайл отправляет `PLAY_VIDEO` обратно в bridge_player.
- **Минимальная сборка**: только декодеры (h264, vp9, av1/libdav1d, opus, aac, mp3, mjpeg, png),
  демультиплексоры (matroska, mov, webm), протоколы (file, pipe, unix, fd) и
  фильтры (aresample, scale, atempo, volume).

---

### 10. `sponsorblock.py` -- Клиент SponsorBlock

Получает сегменты для пропуска из краудсорсингового API SponsorBlock.

- Пропускаемые категории: `sponsor`, `selfpromo`, `interaction`.
- Кэш: `cache/sponsorblock_<videoId>.json`, TTL 24 часа.
- Используются только сегменты с `actionType=skip`.
- Интеграция: bridge_player проверяет каждое обновление POS по списку сегментов.

### 11. `watchstats.py` -- Атрибуция истории просмотра YouTube

Отправляет пинги воспроизведения/просмотра, чтобы просмотренные видео
попадали в историю YouTube и влияли на алгоритм рекомендаций.

**Конвейер:**
1. Запрашивает `/watch?v=<id>` с Firefox-JA3 куками (через `auth_cookies.py`).
2. Извлекает `videostatsPlaybackUrl` и `videostatsWatchtimeUrl` из
   `ytInitialPlayerResponse.playbackTracking`.
3. Генерирует 16-символьный `cpn` (client playback nonce).
4. Начальный пинг `state=playing` при t=0.
5. Пинги heartbeat на 10с, 20с, 30с, затем каждые ~40с с 15% джиттером.
6. Финальный пинг `state=ended` при остановке (если просмотрено >= 8с).

**Безопасность**: только чтение (без лайков/комментариев/подписок). Kill-switch через
переменную окружения `WATCHSTATS_COOKIES=0`. Автостоп при 401/403 (протухшие куки).

### 12. `auth_cookies.py` -- Cookie-аутентификация YouTube

Загружает куки в формате Netscape из `cache/yt_cookies.txt` (экспортированы из
Firefox через `extract_cookies.py`).

- curl_cffi-сессия с TLS-имперсонацией `firefox133`.
- Вычисляет SAPISIDHASH на каждый запрос: `SHA1(timestamp + " " + SAPISID + " " + origin)`.
- Поглощает Set-Cookie ответы обратно на диск для персистентности ротации SID.
- Отправляет только критичные для аутентификации куки (SAPISID, SID, LOGIN_INFO и т.д.) --
  обрезает трекинговый шум (ST-*), чтобы не превысить лимит заголовка Cookie в curl.

---

### 13. `recs_pipeline.py` -- Рендерер рекомендаций для боковой панели

Получает рекомендации watch-next для воспроизводимого видео и рендерит
их как графические тайлы для бокового меню ffplay-yt.

- **Аутентифицированный путь**: TVHTML5 InnerTube `/next` (персонализированные рекомендации).
- **Неаутентифицированный фоллбэк**: WEB-клиент `/next` с публичным API-ключом.
- Рекурсивно извлекает `compactVideoRenderer` и `tileRenderer` из ответа.
- Скачивает превью JPEG и рендерит текстовые полосы (название + канал + просмотры)
  как PNG 456x70 через PIL (шрифты DejaVu Sans).
- Выводит строки `RECS_ITEM <videoId>\t<thumb_path>\t<text_path>` в stdout.
- Поддержка continuation-токенов для бесконечной прокрутки (режим `--continuation`).

### 14. `tui_player.py` -- Proof-of-Concept терминальный видеоплеер

Автономный плеер, декодирующий видео целиком в терминале через kitty graphics protocol.
Не используется в основном конвейере (заменён ffplay-yt).

- ffmpeg декодирует в PNG-кадры на stdout -> Python читает и передаёт через kitty.
- Аудио: ffmpeg -> s16le PCM 48кГц стерео -> aplay (ALSA).
- A/V-синхронизация: wall-clock vs frame_no/fps, пропуск кадров при отставании >200мс.
- Пауза: пробел (SIGSTOP/SIGCONT на процесс aplay).

---

### 15. `stream_player.py` -- Стриминговый плеер на FIFO

Альтернативный конвейер воспроизведения с использованием именованных FIFO вместо растущих файлов.

- SABR-клиент заполняет буферы треков в основном потоке.
- Потоки-писатели сливают `track_state.buf` в `/tmp/yt-stream-<pid>/video.fifo`
  и `audio.fifo` через `os.write()`.
- ffmpeg читает оба FIFO, мультиплексирует в live matroska на stdout.
- ffplay читает stdin. Перемотки нет (только линейное воспроизведение).
- Вытеснен подходом с растущими файлами в bridge_player.py.

## Схема потоков данных

### Аутентификация
```
auth.py (Device Grant)  -->  cache/oauth.json         -->  Bearer-токен InnerTube
extract_cookies.py      -->  cache/yt_cookies.txt     -->  auth_cookies.py (SAPISIDHASH)
```

### Получение ленты
```
InnerTube.home()     -->  TVHTML5 FEwhat_to_watch  -->  4 полки x 5 видео
InnerTube.next(vid)  -->  TVHTML5 /next pivot      -->  ~10 полок x 3 видео
FeedLoader           -->  рекурсивные /next pivot   -->  бесконечная прокрутка
InnerTube.search(q)  -->  TVHTML5 /search          -->  результаты поиска
```

### Видео-стриминг
```
pr_fetch.py (curl_cffi) --> ytInitialPlayerResponse + streamingData
po_token.mjs (bgutils)  --> PO Token, привязанный к контенту
youtubei.js              --> дешифрованный SABR URL (n + sig)
googlevideo SabrStream   --> сырые H.264/Opus fMP4-сегменты
ffmpeg (мультиплексор)   --> растущий .mkv-файл на диске
ffplay-yt (плеер)        --> SDL2-окно с IPC
```

### Кэширование
```
cache/innertube_home.json      -- домашняя лента (TTL 120с)
cache/innertube_next.json      -- /next pivot (TTL 120с)
cache/pr_<vid>.json            -- кэш PR моста (TTL 12ч)
cache/oauth.json               -- OAuth-токены
cache/yt_cookies.txt           -- куки Firefox
cache/strategy_state.json      -- состояние ротации pr_fetch
cache/thumbnails/<vid>.jpg     -- JPEG-превью (бессрочно)
cache/recs_thumbs/<vid>.jpg    -- превью боковой панели
cache/recs_text/<vid>.png      -- текстовые полосы боковой панели
cache/sponsorblock_<vid>.json  -- сегменты SponsorBlock (TTL 24ч)
```

## Ключевые технические решения

1. **Без yt-dlp в пути воспроизведения**: прямой SABR-клиент через npm-пакет googlevideo.
   yt-dlp убран из стриминг-конвейера ради снижения задержки и полного контроля.

2. **Без mpv**: ffplay-yt (патченный ffplay из FFmpeg) -- единственный видеоплеер.
   Кастомный IPC обеспечивает двустороннее управление без оверхеда JSON IPC mpv.

3. **Без постоянного браузера**: PO Token чеканятся через bgutils-js + jsdom
   (BotGuard JS исполняется в headless DOM). Без Camoufox, без Playwright в рантайме.

4. **H.264 вместо VP9**: H.264 декодируется в 2-3 раза быстрее программно, оставляя
   запас для atempo-ускорения и снижая нагрузку на CPU при стриминге.

5. **Архитектура растущего файла**: ffmpeg пишет matroska в файл, ffplay читает его
   по мере роста. Перемотка за буфер запускает новую сессию (новый файл + `OPEN` по IPC).
   Превосходит пайп-подходы по поддержке перемотки.

6. **Липкая ротация стратегий**: 24 комбинации (TLS x прокси x эндпоинт) с ротацией
   при блокировке и автоматическим перезапуском раунда при исчерпании всех.
   Выдерживает адаптивную бот-детекцию YouTube между сессиями.

7. **Cookie-based watchstats**: отправляет настоящие пинги `/api/stats/watchtime`
   с Firefox-куками + SAPISIDHASH, чтобы видео попадали в историю YouTube.

8. **Kitty graphics protocol**: терминальные изображения только PNG (f=100), JPEG-превью
   конвертируются через PIL с быстрым сжатием (level 1).

## Зависимости

**Python 3.11** (venv в `.venv/`):
- `curl_cffi` -- реальные TLS-отпечатки браузеров для запросов /watch
- `httpx` -- HTTP/2 клиент для InnerTube, SponsorBlock, превью
- `Pillow` -- конвертация JPEG->PNG, рендеринг текста, блендинг превью

**Node.js** (локальные `node_modules/`):
- `youtubei.js` -- дешифровка n-параметра, расшифровка подписей
- `googlevideo` -- SabrStream, клиент протокола SABR
- `bgutils-js` -- исполнитель BotGuard-челленджей, чеканка PO Token
- `jsdom` -- DOM-окружение для интерпретатора BotGuard
- `canvas` -- зависимость jsdom для рендеринга DOM

**Системные утилиты:**
- `ffmpeg` / `ffprobe` -- обработка медиа (системная установка)
- `ffplay-yt` -- кастомная сборка ffplay в `ffplay-yt/bin/ffplay-yt`
- `xdotool` / `wmctrl` -- управление фокусом окон X11
- `aplay` -- ALSA-вывод звука (используется только tui_player.py)
- терминал kitty (обязателен для поддержки graphics protocol)
